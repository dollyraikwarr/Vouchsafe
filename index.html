The problem

Getting paid for technical work still runs on trust.

🧾 Chasing invoices

Work ships, payment doesn't. Net-30 quietly becomes net-never, and there's no leverage left once the deliverable is already out the door.

💸 Escrow that prices out small jobs

Traditional escrow fees make sense on a $10,000 contract and make no sense on a $150 bug fix — so the small jobs just go unprotected.

🗂️ No record either side can point to

When a dispute happens, it's one person's memory of a Slack thread against another's — nothing either party can independently verify.

How it works

Four calls. One state machine. No custody outside the contract.

01

Create

The client defines the developer, the payment token, the amount, and a deadline. Terms are recorded on-chain before anything is funded.

create_engagement()
02

Fund

Payment transfers from the client's wallet into the contract's escrow — held by code, not by either party, until the work is approved.

fund_engagement()
03

Submit

The developer attaches proof of the delivered work — URL, PR link, commit hash, and a note — directly to the on-chain engagement record.

submit_work()
04

Approve

The client approves. In the same transaction, the contract releases the escrowed payment to the developer and closes the engagement.

approve_work()
The guarantee

Funds cannot move without approval. Full stop.

Release attempt · before client approval
approve_work(482)
✕ BLOCKED

The contract requires the engagement to be in WorkSubmitted and the caller to be the client. No admin key, no override, no exceptions.

Release attempt · after client approval
approve_work(482)
✓ RELEASED — 500 XLM → developer

Same function call, different contract state. The contract decides when funds move — not either party, and not a database either side controls.

Protocol

Every engagement is a permanent on-chain record.

Client, developer, amount, token, status, and the developer's submitted deliverable — URL, PR, commit hash, and note — are all readable straight off the ledger through get_engagement(). Nothing about what was promised, paid, or delivered lives in a database either side controls.

Architecture

Three layers. No custody outside the contract.

Client & developer layer

Wallet connect via Stellar Wallets Kit. The browser signs; keys never leave it.

Contract layer

EscrowEngagement — Soroban / Rust. create → fund → submit_work → approve_work

Chain layer

Stellar Testnet — Soroban runtime, Horizon + RPC.

Soroban Rust Stellar Wallets Kit Stellar SDK Vanilla JS — no bundler
