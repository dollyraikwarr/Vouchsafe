Problem

How it works

Four steps. One state machine.

01 📝

Create

The client defines the engagement — developer address, payment amount, token, and deadline.

02 💵

Fund

Payment locks into the contract's on-chain escrow. Nobody can spend it until the state machine moves.

03 📤

Submit

The developer records deliverable proof on-chain: a URL, a PR link, a commit hash, and a note.

04 ✅

Approve

Client approves. Payment releases to the developer automatically, in the same transaction.

On-chain record

Every engagement is a permanent, queryable record

get_engagement() returns exactly what happened — client, developer, amount, deliverables, and whether payment actually released. No summary, no interpretation.

FAQ

Straight answers

What happens if the client doesn't approve? +
If a client fails to approve, the escrowed funds remain locked in the contract. A dispute resolution or arbitration layer is planned for the next roadmap phase to handle contested deliverables.
What assets can I use? +
You can use any standard Stellar asset (Stellar Asset Contract / SAC) for escrows. This includes native XLM as well as custom tokens and stablecoins.
Is it secure? +
Yes, all state transitions and escrow releases are governed strictly by the smart contract code compiled to WASM and verified on the Stellar ledger. Authorization checks ensure only the developer can submit work, and only the client can approve it.
What format is required? +
When submitting work, the developer must provide a deliverable URL (e.g. GitHub link), a pull request link, a commit hash, and a short completion note.
